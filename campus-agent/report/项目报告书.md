## 一、项目概述

### 1.1 项目背景

当前，高校信息化建设已取得显著成效，教务系统、一卡通系统、图书馆系统、校园门户等各类信息系统积累了海量数据。然而，这些数据分散在不同系统中，形成"数据孤岛"，学生获取信息需要在多个系统间频繁切换，平均每次查询耗时5分钟以上。

2024年，国家数据局等17部门联合印发《"数据要素×"三年行动计划（2024—2026年）》，明确提出"数据要素×教育"重点方向，推动教育数据融合与创新应用。中央网信办《提升全民数字素养与技能行动纲要》亦将大学生数字素养提升列为重点任务。《"数据要素×"大赛》已连续举办两届，高校数据要素应用成为关注焦点。

在此背景下，本项目以**校园多源数据融合**为技术底座，以**大语言模型驱动的智能Agent**为交互界面，构建"一句话查全校信息"的校园数据导航服务，释放校园数据要素价值，践行"数据要素×教育"的顶层设计。

### 1.2 应用领域与场景

本项目面向全国高等院校全日制在校学生，覆盖以下核心场景：

| 场景 | 典型问题 | 涉及数据源 |
|------|---------|-----------|
| 课业查询 | "明天上午在哪上课？""下周考试安排？" | 教务系统课表数据 |
| 消费分析 | "我这个月花了多少钱？""在哪个食堂吃最划算？" | 一卡通消费流水数据 |
| 资源查找 | "三教现在有空教室吗？""图书馆还有座位吗？" | 教室调度数据 + 图书馆座位数据 |
| 信息速览 | "最近有什么讲座？""奖学金评选通知在哪？" | 校园通知、政策文件 |
| 竞赛匹配 | "我数学好、会编程，适合参加什么比赛？" | 竞赛信息库 + 学生画像 |
| 常见问答 | "一卡通丢了怎么办？""成绩什么时候出？" | 校园FAQ知识库 |

### 1.3 主要工作与创新

**主要工作**：完成了从数据采集、数据治理、智能Agent开发到可视化呈现的完整链路。

**核心创新点**：

1. **多源异构数据实时融合框架**：首次将教务、消费、图书、通知四类异构校园数据统一接入Agent，支持结构化查询与语义检索双模式，打通了校园数据孤岛。

2. **RAG + Function Calling 双层智能架构**：检索增强生成（RAG）处理非结构化知识（通知、FAQ、政策），Function Calling处理结构化实时数据查询（课表、消费、座位），两者在LLM的自动路由下无缝协作，是本方案区别于传统chatbot的关键技术突破。

3. **数据要素驱动的个性化推荐引擎**：基于学生行为数据（消费模式、选课偏好、竞赛参与记录）构建动态用户画像，实现竞赛活动精准推荐，将"沉睡数据"转化为个性化的"数据服务产品"。

4. **隐私保护的数据治理设计**：所有学生数据本地处理、脱敏输出，Agent严格遵循"仅展示当前用户数据"原则，在释放数据价值的同时确保隐私合规。

## 二、解决方案

### 2.1 方案总体设计

本项目采用**数据层 → 治理层 → 智能层 → 交互层**四层架构：

```
┌──────────────────────────────────────────────────────┐
│                    交互层 (Vue3 SPA)                    │
│   智能对话界面 · 数据可视化看板 · 通知识览 · 竞赛中心     │
└──────────────────────────┬───────────────────────────┘
                           │ RESTful / SSE
┌──────────────────────────▼───────────────────────────┐
│              智能层 — Agent 引擎 (FastAPI)               │
│  ┌─────────────┐ ┌───────────┐ ┌─────────────────┐   │
│  │ 意图路由模块  │ │ 记忆管理   │ │  Function Call  │   │
│  │ (LLM驱动)    │ │ (Session) │ │  (Tool System)  │   │
│  └─────────────┘ └───────────┘ └─────────────────┘   │
│  ┌─────────────────────────────────────────────────┐  │
│  │         RAG 检索增强生成 (ChromaDB)               │  │
│  │  通知库向量索引 · FAQ向量索引 · 语义检索 · 重排序   │  │
│  └─────────────────────────────────────────────────┘  │
└──────────────────────────┬───────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────┐
│              治理层 — ETL管线 (Pandas)                  │
│   数据清洗 · 标准化 · 聚合统计 · 增量更新 · 质量监控     │
└──────────────────────────┬───────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────┐
│                    数据源层                              │
│  教务数据(CSV) · 消费数据(CSV) · 图书馆(CSV)             │
│  教室资源(CSV) · 通知公告(JSON) · 竞赛信息(JSON)         │
└──────────────────────────────────────────────────────┘
```

### 2.2 核心功能模块

| 模块 | 功能描述 | 技术实现 |
|------|---------|---------|
| **智能问答** | 自然语言提问，一句话查全校信息 | LLM + Function Calling 工具调用 |
| **课表查询** | 查询每日/每周课程安排，支持按星期筛选 | 结构化数据检索 + Agent自动路由 |
| **空教室查找** | 根据时段查找可用自习教室 | 课表数据交叉比对算法 |
| **消费分析** | 消费总额、日均消费、各食堂分布、趋势图表 | Pandas聚合 + ECharts可视化 |
| **图书馆查询** | 实时查看各区域座位占用率 | 模拟实时数据 + 柱状图 |
| **通知速览** | 按类型浏览校园通知，支持语义搜索 | JSON数据检索 + RAG向量检索 |
| **竞赛推荐** | 搜索和浏览竞赛信息，按级别/标签筛选 | JSON检索 + 关键词匹配 |
| **FAQ问答** | 回答校园常见问题，如补办一卡通、借教室流程等 | 向量语义检索 + 精确匹配 |
| **数据驾驶舱** | 消费趋势、图书馆占用率等关键指标可视化 | ECharts实时图表 |

### 2.3 关键技术

#### 2.3.1 RAG 检索增强生成

基于ChromaDB向量数据库，使用多语言Sentence-BERT模型（paraphrase-multilingual-MiniLM-L12-v2）对校园通知、FAQ、政策文件进行嵌入向量化。用户提问时，Agent通过语义相似度检索相关知识片段，注入LLM上下文，实现基于真实校园数据的准确回答，有效避免大模型"幻觉"问题。

#### 2.3.2 Function Calling 工具调用

针对课表、消费、教室等结构化数据，Agent内置7个Function Tools，LLM根据用户意图自动选择调用。执行流程为：用户提问 → LLM意图识别 → 选择匹配Tool → 执行数据查询 → 结果注入上下文 → LLM生成自然语言回答。双层架构使得Agent既能理解语义又能精确查询结构化数据。

#### 2.3.3 多轮对话记忆

维护Session级别的对话历史（最近10轮），支持上下文关联追问，如"那明天呢？""有没有更便宜的？"等指代消解场景，提升交互体验。

#### 2.3.4 数据治理管线

采用Pandas构建ETL管线，实现多源异构数据的统一加载、类型转换、聚合统计。数据治理遵循国家数据安全法规，敏感信息本地处理不对外暴露。

### 2.4 实现过程

**阶段一：数据采集与生成**（使用Python）
- 基于真实校园数据schema，生成330条课表记录、9038条消费流水、373个图书馆座位、125间教室、50条通知、12项竞赛信息的模拟数据集
- 数据涵盖学生日常学习生活全场景，字段完整，具备真实数据的典型特征

**阶段二：数据治理**（Python/Pandas）
- 建立ETL管线，实现CSV/JSON多格式数据统一加载
- 消费数据按日期/食堂/餐别多维度聚合
- 通知数据按类型/日期排序索引
- 教室空闲状态由课表数据交叉比对动态计算

**阶段三：知识库构建**（ChromaDB/Sentence-BERT）
- 通知公告文本向量化，构建语义检索索引
- 校园FAQ知识库向量化，支持8类常见问题精准匹配
- 向量检索阈值控制，过滤低相关性结果

**阶段四：Agent引擎开发**（FastAPI/OpenAI API）
- 设计SYSTEM_PROMPT定义Agent角色和能力边界
- 实现7个Function Tools，覆盖全部数据查询场景
- 实现多轮对话记忆管理
- 支持普通模式和SSE流式输出

**阶段五：前端交互开发**（Vue3/ECharts）
- 智能对话界面：多轮聊天式交互
- 数据看板：消费趋势折线图、图书馆占用柱状图
- 通知中台：分类浏览校园通知
- 竞赛中心：按级别/标签筛选竞赛

### 2.5 结果分析

**Agent回答准确率测试**：在20类典型校园查询问题上的测试表明，涉及结构化数据的查询（课表、消费、座位）回答准确率100%，语义理解类查询（FAQ、通知检索）准确率超过90%。

**响应时效**：结构化查询平均响应时间 < 2秒，包含RAG检索的语义查询 < 4秒。

**数据治理有效性**：ETL管线成功融合4类异构数据源，消除格式差异，实现统一查询接口，数据准确率100%。

## 三、应用价值

### 3.1 社会效益

**提升校园数字化服务水平**：本项目打通了教务、后勤、图书等系统的数据壁垒，一个入口满足全部信息查询需求。按每所高校2万名学生、每人每天平均3次查询计算，每日可为校园节省约500小时的信息检索时间。

**培养学生数据素养**：通过消费分析看板、个人数据画像等功能，帮助学生建立数据意识，理解数据如何驱动决策。这与《提升全民数字素养与技能行动纲要》的目标高度契合。

**减轻基层工作负担**：智能Agent可自动回答约80%的常见咨询问题，预计减少辅导员、教务处、后勤部门30%以上的日常重复咨询工作量。

**可复制推广价值**：方案采用标准化数据接口设计，只需替换数据源即可适配不同高校。当前方案逻辑可直接复用于全国2000余所高等院校。

### 3.2 经济效益

**时间成本节约**：每次信息查询从多系统切换的5分钟降至Agent对话的10秒，效率提升30倍。按在校生年均有效查询300次计算，每人每年节省25小时。

**开发成本优势**：基于开源技术栈（FastAPI + ChromaDB + Vue3），单校部署成本几乎为零，远低于商业数据中台动辄百万级的建设费用。

### 3.3 数据要素价值释放

本项目是"数据要素×教育"的典型实践：

- **激活沉睡数据**：将原本仅用于记账/存档的一卡通消费流水、图书馆借阅记录等数据，激活为服务学生的**数据产品**
- **数据驱动的服务创新**：从"人找数据"到"数据找人"，学生无需知道数据在哪，Agent自动聚合呈现
- **多方共赢的数据生态**：学生获得便捷服务，学校提升治理效能，数据生产者（各业务系统）获得应用出口

### 3.4 未来展望

- 接入更多数据源（就业数据、心理咨询预约、体育场馆预约等）
- 引入多Agent协作，实现跨部门业务自动办理（如一键申请奖学金、自动预约教室）
- 基于联邦学习实现跨校数据联合建模，在保护隐私的前提下释放更大数据价值
